# Bug sweep — full-repo pass (2026-07-24)

Follow-up to the audio-cutout investigation (PR #5): a systematic read of the
sim core, run layer, controller, persistence, data tables, and UI looking for
anything else broken. Findings below, split into **fixed** and
**audited — judged fine / noted**.

## Fixed in this PR

### 1. Stale `combo_bonus` after a charge reroll (sim, gameplay correctness)
`lib/sim/combat.dart` — `combatReroll` (the relic-charge `reroll {die}`
command) changed a rolled face **without re-detecting combos**, while `roll`
and `reroll_risky` both re-detect. Combos are contractually "a pure function
of the rolled pool" (m4 §3), so the state went inconsistent the moment a
charge reroll changed the combo picture:

- rerolling one die of a pair **kept paying the +1 pair bonus on both dice**
  (including the rerolled one) for the rest of the turn;
- rolling INTO a new pair paid nothing;
- a new triple never ignited; a new straight never earned the free reroll.

Only reachable with Gambler's Eye / Twin Eye (or the Gambler's start relic),
which is why it survived: the autoplay bot never issues `reroll`.

**Fix:** `combatReroll` now calls `_detectAndApplyCombos` after the reroll,
exactly like the risky path. The once-per-turn guards (`ignited`,
`free_reroll_next`) already make re-detection idempotent, so nothing can
double-apply. The combat screen's charge-reroll button now also announces the
resulting combo events (it silently ignored them before).
`docs/m4-sim-contract.md` §1/§3 updated to match. Regression test added
(`charge reroll re-detects combos`).

**Determinism:** no RNG consumption changed (detection consumes zero RNG).
Gates measured byte-identical against main (post-#10, SIM_VERSION 6) —
golden(20260723) = `1842571558`, 200-seed autoplay 67.0% / 0 invalids /
0 twin fails, all matching main's own baseline — so **no SIM_VERSION bump**,
per the same convention as the zombie-win fix. Runs that actually use charge
rerolls can produce different (now correct) events; old replay logs of such
runs would diverge, but nothing in the shipped product replays cross-version
command logs.

### 2. Death-screen insight that lies about the boss (content, §Ethics)
`lib/data/insights.dart` — the boss bucket still said *"The Tyrant's turn 4
hits for 25"*. Since the v0.3.0 rebalance the Tyrant's turn-4 attack is
**32** base (more on hard/ascension). A coaching line that understates the
hit by 7+ is worse than none — the fair-death pillar says every insight is
"concrete and honest". Reworded number-free (*"turn 4 is its heaviest hit"*)
so it stays true across difficulty/ascension. The other two boss lines were
checked against the actual pattern and are correct.

### 3. Leftover editing junk in an event label (content)
`lib/data/events.dart` — Ember Moths option read
*"Let them feed (-8 embers... wait, +8 embers)"*. Authoring leftover shipped
to players; now *"Let them feed (+8 embers)"*. Labels are not hashed, so this
is presentation-only.

### 4. Map marker "walks" in from the previous run (UI)
`lib/ui/screens.dart` — `_MapScreenState._walkFrom` is deliberately `static`
so the marker walks node-to-node across map visits, but it was never reset
between runs. Node ids restart at 1 every run, so after finishing a run at
node N, the next run's first map view animated the delver walking from
old-run node N across the new map. Now keyed by the run seed (new
`GameController.runSeed` getter) and reset when it changes.

### 5. Unclamped volumes from a corrupt settings file (robustness)
`lib/audio/settings.dart` — `AudioSettings.fromJson` accepted any number; an
out-of-range `musicVolume`/`sfxVolume` (hand-edited or corrupt file) would
crash the Settings screen (Flutter's `Slider` asserts `value ∈ [min, max]`).
Values are clamped to `[0, 1]` on load.

### 6. Wrong stream documentation for event grants (docs)
`lib/data/events.dart` header claimed `gain_random_die` / `gain_random_relic`
draw from the **shuffle** stream; `lib/sim/run_layer.dart`'s header said the
same. The code (behavior-anchored by the golden) draws grants from **loot**
and only the die **loss** from shuffle. Comments corrected — code is the
anchored truth. Also documented the previously-undocumented `gold_after`
effect (Gambler's Table payout).

## Audited — judged fine / noted (no change)

- **Risky reroll + `on_max_gold`:** a natural max face still pays the relic
  gold even when the −1 pip penalty drops the effective value below max
  ("shows its highest face" is literally true; `on_max_bonus` correctly does
  NOT apply). Deliberate reading; changing it would shift RNG-visible economy.
- **`meta.bestAscension` semantics:** stores "next rung unlocked" (win at 0 →
  1), and the Ledger shows it as "Best ascension". Slightly generous label;
  product call, left as is.
- **First-run easy steering** persists `preferredDifficulty='easy'` after the
  first run even if the selector was never tapped — consistent with "what's
  highlighted is what they get".
- **`delveAgain` after a Daily** starts a fresh normal-difficulty run
  (daily runs are pinned to normal) rather than the player's sticky
  preference. Defensible ("same run, new seed"); noted only.
- **Boss/elite name-plate splash** replays on app-restore into a boss fight.
  Cosmetic; arguably a feature.
- **Choreography vs `terminalHold`:** attack path ≤ ~1260 ms < 1300 ms hold;
  end-turn death path ≤ ~1360 ms < 1450 ms hold. Timing budget holds.
- **Daily Delve uses device-local date** — already on the v0.4 backlog.
- **`Rng.range` modulo bias** — inherent to the golden-anchored Lua-port LCG;
  cannot change without re-anchoring everything.
